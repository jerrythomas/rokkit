# Theming & Design

Rokkit separates **layout CSS** (structural) from **theme CSS**
(visual). Components ship unstyled with `data-*` attribute
hooks; themes provide the visual layer — colors, radii, shadows —
without touching structure.

## Skins, styles, modes

Three orthogonal dimensions:

| Dimension    | Drives                                                         | Default   |
| ------------ | -------------------------------------------------------------- | --------- |
| `data-style` | Visual character (`rokkit`, `minimal`, `material`, `frosted`, `zen-sumi`) | `rokkit`  |
| `data-skin`  | Palette mapping (roles → tailwind colours)                     | `default` |
| `data-mode`  | Light / dark                                                   | `light`   |

All three are attributes on `<html>`. Flipping any one re-skins
the entire app instantly — no rebuild.

```html
<html data-style="zen-sumi" data-skin="default" data-mode="dark"></html>
```

## Theme imports

Each style ships its own CSS file. Import the base + the style
you want as the default; lazy-load others if you support runtime
switching:

```css
@import '@rokkit/themes/base.css';
@import '@rokkit/themes/rokkit.css';
```

## Semantic colour tokens

Themes resolve to a fixed vocabulary of 24 named tokens used by
components. These are the CSS custom properties emitted by
`presetRokkit()` when `tokens: 'core'`:

**Surface**
`--paper` / `--paper-soft` / `--paper-mute` / `--paper-edge`

**Text + lines**
`--ink` / `--ink-mute` / `--ink-soft` / `--ink-faint`

**Brand**
`--primary` / `--on-primary` / `--accent` / `--accent-soft`

> Note: `secondary` is a _skin-config role_ (an input to the
> preset) — it is **not** a named CSS variable that components
> reference. Use `--primary` / `--accent` for brand colour in
> your custom CSS.

**Status** (each has a `-soft` tint variant)
`--success` / `--success-soft` · `--warning` / `--warning-soft` ·
`--danger` / `--danger-soft` · `--error` / `--error-soft` ·
`--info` / `--info-soft`

**Utility**
`--focus-ring` / `--shadow-tint`

Each token auto-inverts for dark mode without explicit
`@media (prefers-color-scheme)` rules.

## Data-attribute styling

Components don't use BEM-style class names — they expose state
through `data-*` attributes:

```css
[data-list-item][data-active='true'] {
  box-shadow: inset 2px 0 0 0 var(--accent);
}

[data-button][data-loading='true'] {
  opacity: 0.6;
}
```

Override by selecting on those attributes in your own CSS. Don't
fight component classes — there usually aren't any.

## Roles + Palettes

Build a custom skin by mapping the semantic roles to Tailwind
palette names. The CLI generates this for you — run
`npx @rokkit/cli init` and pick a preset. The canonical
shape uses palette-name strings per role:

```js
// rokkit.config.js — generated by `npx @rokkit/cli init`
export default {
  skin: {
    surface: 'slate',   // same palette for light + dark
    ink: 'slate',
    primary: 'orange',
    accent: 'sky',
    success: 'green',
    warning: 'yellow',
    danger: 'red',
    error: 'red',
    info: 'cyan'
  },
  colorSpace: 'rgb',
  tokens: 'core'
}
```

For a dual-palette role (different palette per mode), use the
`{ light, dark }` object form — e.g. the zen-sumi starter:

```js
skin: {
  surface: { light: 'kami', dark: 'sumi' },
  ink:     { light: 'kami', dark: 'sumi' },
  primary: 'shu',
  accent:  'shu'
}
```

The `skin list` and `skin create` CLI commands manage skin
entries in your `rokkit.config.js` interactively.

## Runtime switching

### Prevent SSR flash

Add the `themeHook` to `src/hooks.server.ts`. It injects a
synchronous inline script that reads persisted settings from
`localStorage` before first paint — preventing a flash of the
wrong style or mode:

```ts
// src/hooks.server.ts
import { themeHook } from '@rokkit/unocss/hooks'
export const handle = themeHook({ defaultMode: 'system' })
```

`defaultMode: 'system'` resolves the OS preference when nothing
is persisted. Compose with other handlers via `sequence()` from
`@sveltejs/kit/hooks`.

### Theme toggle component

`@rokkit/app` ships a ready-made three-way toggle:

```svelte
<script>
  import { ThemeSwitcherToggle } from '@rokkit/app'
</script>

<ThemeSwitcherToggle />
```

### Writing to the vibe store

For programmatic switching, import `vibe` from `@rokkit/states`
and assign to its property setters:

```ts
import { vibe } from '@rokkit/states'

vibe.style   = 'zen-sumi'  // switch visual style
vibe.mode    = 'dark'      // 'light' | 'dark'
vibe.density = 'compact'   // 'compact' | 'cozy' | 'comfortable'
```

Changes are reactive — any component reading `vibe.style` or
`vibe.mode` re-renders automatically.

## When to roll your own theme

Style CSS files are themable in isolation — copy one of the five
shipped styles, swap the surface treatments and chrome, and you
have a brand-aligned variant without touching component code.
The data-attribute selectors stay the same; only the visual
recipe changes.

For the interactive editor, see the
[Theme Wizard](/app/theming) demo — it lets you preview palette
changes in real time and export a `tokens.css` you can drop
into your app.
