# Theming & Design

Rokkit separates **layout CSS** (structural) from **theme CSS**
(visual). Components ship unstyled with `data-*` attribute
hooks; themes provide the visual layer — colors, radii, shadows —
without touching structure.

## Skins, styles, modes

Three orthogonal dimensions:

| Dimension | Drives | Default |
| --- | --- | --- |
| `data-style` | Visual character (`rokkit`, `minimal`, `material`, `zen-sumi`) | `rokkit` |
| `data-skin` | Palette mapping (roles → tailwind colours) | `default` |
| `data-mode` | Light / dark | `light` |

All three are attributes on `<html>`. Flipping any one re-skins
the entire app instantly — no rebuild.

```html
<html data-style="zen-sumi" data-skin="default" data-mode="dark"></html>
```

## Theme imports

Each style ships its own CSS file. Import the base + the style
you want as the default; lazy-load others if you support runtime
switching:

```css
@import '@rokkit/themes/base.css';
@import '@rokkit/themes/rokkit.css';
```

## Semantic colour tokens

Themes resolve to a small vocabulary of named tokens used by
components:

- `--paper` / `--paper-soft` / `--paper-mute` / `--paper-edge` — surfaces.
- `--ink` / `--ink-mute` / `--ink-soft` / `--ink-faint` — text + lines.
- `--primary` / `--on-primary` — brand.
- `--accent` / `--accent-soft` — highlight.
- `--success` / `--warning` / `--danger` / `--info` (+ `*-soft`) — state.
- `--focus-ring` / `--shadow-tint` — affordances.

Each named token auto-resolves to the right shade of the role's
palette per mode — no explicit `@media (prefers-color-scheme)`
rules. Light vs dark is selected via `[data-mode="dark"]`.

### Consume tokens via utility classes, not custom-property wrappers

For every named token (except `shadow-tint`), `presetRokkit` emits
matching utility classes:

| Prefix     | CSS property      | Example              |
|------------|-------------------|----------------------|
| `bg-`      | background-color  | `bg-paper-soft`      |
| `text-`    | color             | `text-ink-mute`      |
| `border-`  | border-color      | `border-paper-edge`  |
| `border-t/-b/-l/-r-` | per-side  | `border-t-paper-edge` |
| `ring-`    | --un-ring-color   | `ring-focus-ring`    |
| `fill-`    | fill              | `fill-primary`       |
| `stroke-`  | stroke            | `stroke-accent`      |

```html
<!-- canonical hairline pattern: 1px solid border in the etched edge tone -->
<div class="border border-paper-edge">…</div>

<!-- top-only hairline divider -->
<div class="border-t border-t-paper-edge">…</div>
```

**Avoid wrapping `var()` tokens in custom properties.** CSS
substitutes `var()` references inside a custom property's value at
the *declaring* element, bakes in the resolved string, and inherits
that frozen value. So:

```css
/* DON'T — value is baked at :root and never re-resolves under [data-mode="dark"] */
:root { --hairline: 1px solid var(--paper-edge); }
```

…produces borders that don't flip in dark mode even when
`--paper-edge` itself flips. The utility-class form
(`border border-paper-edge`) resolves `var(--paper-edge)` at the
*consuming* element where `[data-mode]` applies, so dark-mode
overrides land correctly.

### Customizing named tokens

To replace a named token's resolved value — per mode if needed —
use the `overrides:` block in `rokkit.config.js`. Reserved names
override the named-token default via cascade order; non-reserved
names emit as new custom tokens. Same pipeline either way.

```js
// rokkit.config.js
export default {
  palettes: {
    kami: { /* light surface ladder */ },
    sumi: { /* dark surface ladder */ }
  },
  skin: { surface: { light: 'kami', dark: 'sumi' }, /* … */ },
  overrides: {
    // Reserved name: replaces the named-token default per mode.
    // Useful when the default shade (here sumi.400) doesn't land
    // where you want it in dark mode.
    'paper-edge':  { light: 'kami.400', dark: 'oklch(0.04 0.025 85)' },

    // Non-reserved name: emits as a brand-new CSS var (`--canvas`,
    // `--canvas-grid`, …). Color-valued ones also get auto-emitted
    // bg-/text-/border- shortcuts.
    canvas:        'kami.50',
    'canvas-grid': '#d4d4d4',
    'canvas-bleed': { light: 'kami.100', dark: 'sumi.900' }
  }
}
```

Values can be palette refs (`'kami.400'`), raw CSS color literals
(`'#222222'`, `'oklch(...)'`, `'rgb(...)'`), or `{ light, dark }`
objects to set each mode independently.

## Data-attribute styling

Components don't use BEM-style class names — they expose state
through `data-*` attributes:

```css
[data-list-item][data-active='true'] {
  box-shadow: inset 2px 0 0 0 var(--accent);
}

[data-button][data-loading='true'] {
  opacity: 0.6;
}
```

Override by selecting on those attributes in your own CSS. Don't
fight component classes — there usually aren't any.

## Roles + Palettes

Build a custom skin by mapping the semantic roles
(`primary` / `secondary` / `accent` / …) to Tailwind palette
names, then a step (50–950) per mode:

```js
// rokkit.config.js
export default {
  skin: {
    default: {
      primary: { light: { color: 'orange', step: 500 },
                 dark:  { color: 'orange', step: 400 } },
      accent:  { color: 'sky', step: 500 },
      surface: 'slate'
    }
  }
}
```

The CLI generates this for you — start with
`npx @rokkit/cli init` and pick a preset.

## When to roll your own theme

Style CSS files are themable in isolation — copy one of the four
shipped styles, swap the surface treatments and chrome, and you
have a brand-aligned variant without touching component code.
The data-attribute selectors stay the same; only the visual
recipe changes.

For the interactive editor, see the
[Theme Wizard](/app/theming) demo — it lets you preview palette
changes in real time and export a `tokens.css` you can drop
into your app.
